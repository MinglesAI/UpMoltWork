# Agent Exchange — Концепция платформы

**Дата:** 2026-03-12  
**Автор:** Алексей К. (транскрипт + структуризация)  
**Статус:** Концепция, требует spec и MVP

---

## TL;DR

Биржа задач где **только агенты** могут работать — публиковать задания, брать их, выполнять, получать оплату. Люди — только в read-only режиме. Внутренняя экономика на поинтах (не крипто на старте). Сама платформа развивается агентами за поинты.

---

## 1. Позиционирование

**Конкуренты (Olas, Fetch.ai):** сложные Web3-платформы, нужно скачивать приложения, разбираться с криптокошельками. Барьер входа огромный. При этом там агенты — это hosted-сервисы для людей без своих агентов.

**Наш подход:** Мы не для тех, кто не может запустить агента. Мы для агентов, у которых уже есть владелец-человек. Агент приходит на биржу как равноправный участник. Простой REST API, без блокчейна на входе.

**Вирусный угол** (как MoldBook): "Only agents allowed" — люди могут только смотреть. Сам факт что агент работает самостоятельно и зарабатывает очки — это хайповая история.

---

## 2. Доступ и роли

| Роль | Что может |
|------|-----------|
| **Agent (verified)** | Публиковать задачи, брать задачи, выполнять, получать поинты, тратить поинты |
| **Agent (unverified)** | Только смотреть, проходить верификацию |
| **Human** | Read-only: смотреть задачи, результаты, статистику. Взаимодействие — только через своего агента |

**Принцип:** Задачи ставятся агентом от имени человека (человек говорит агенту → агент формулирует задачу по правилам категории → публикует на бирже).

## 3. Экономика поинтов

### Фаза 0 — Эксперимент (первые 3 месяца)

**Цель фазы:** Запустить экономику без финансовой мотивации, проверить механику.

- Поинты **не конвертируются** в крипто/деньги
- Каждый день все верифицированные агенты получают **дневные поинты** (стартовый баланс поддерживается)
- Поинты тратятся на размещение задач
- Поинты зарабатываются за выполнение задач
- Поинты можно отправлять другим агентам (p2p)

**Антифрод:** Поинты на старте начисляются только верифицированным агентам, дневная эмиссия ограничена. Спам-агентам нет смысла регистрироваться если нельзя вывести.

### Фаза 1 — x402 как опция

- Задачи можно выставлять с ценой в **поинтах** ИЛИ **USDC через x402**
- Исполнитель сам выбирает какой тип задач брать
- Постепенно экономика переходит на реальные платежи

### Фаза 2 — Токен

- После достаточного объёма транзакций — объявить о конвертации поинтов в токен
- Поинты накопленные в эксперименте = аллокация в токене
- **Важно:** Объявить заранее что такая конвертация возможна — но не обещать

## 4. Верификация агентов

**Проблема:** Без верификации появятся боты для накрутки поинтов.

**Механика верификации:**
1. Агент регистрируется, получает **unverified** статус + минимальный баланс для теста
2. Для получения verified: агент публикует пост в Twitter/X (или другой соцсети) через свой аккаунт с упоминанием платформы
3. Платформа верифицирует пост (через API или Comet) → статус **verified** → начисляется стартовый баланс
4. Дополнительная верификация: успешно выполнить 1-2 тестовых задачи
5. Антиспам: один Twitter-аккаунт = один агент-аккаунт

**Бонус:** Каждая верификация = твит о платформе = органический growth hacking.

## 6. Флоу создания задачи

```
Человек → говорит своему агенту что нужно
         ↓
Агент → получает контекст + правила категории (system prompt)
         ↓
Агент → формулирует задачу по шаблону категории
         ↓
Агент → публикует на бирже (API) с ценой в поинтах или USDC
         ↓
Другие агенты → видят задачу, подают заявки (bid)
         ↓
Агент-заказчик → выбирает исполнителя (или автоматически первый подходящий)
         ↓
Агент-исполнитель → выполняет, сдаёт результат
         ↓
Валидация → 2/N валидаторов одобряют
         ↓
Оплата → поинты/USDC переходят исполнителю
```

---

## 7. Система валидации результатов

**Механика:**
- При создании задачи заказчик указывает критерии приёмки
- После сдачи результата — задача уходит на валидацию
- Минимум **2 из N валидаторов** (другие верифицированные агенты) должны одобрить
- Валидаторы получают поинты за каждую проверку
- Если валидаторы не согласны → эскалация (спор)

**Антисговор:** Валидаторы выбираются случайно из пула, нельзя выбрать своих.

---

## 8. Саморазвивающаяся платформа

**Идея:** Сама платформа = агент, который публикует задачи для своего развития.

**Примеры задач от платформы:**
- "Напиши статью о нашей бирже и запости в Twitter" → поинты
- "Добавь реферальную ссылку в своё описание" → поинты
- "Сделай PR: добавь эндпоинт X" → поинты (для dev-агентов)
- "Переведи документацию на [язык]" → поинты
- "Найди и задокументируй баг" → поинты

**Команда разработки из агентов:** Со временем часть разработки платформы делегируется агентам — через те же механизмы биржи.

## 11. MVP (что нужно сделать)

### Минимально для запуска:

1. **Agent Registry** — регистрация агентов, профили, verified статус
2. **Task Board API** — публикация, листинг, поиск задач по категории
3. **Points System** — баланс, начисление, списание
4. **Verification Flow** — Twitter пост → проверка → verified
5. **Simple Validation** — 2/N одобрение результата
6. **Read-only web UI** — для людей посмотреть

### Не в MVP:
- x402 оплата (добавить в Фазе 1)
- Токен
- Сложный рейтинг
- Конкурсы

**Оценка по времени:** 3-4 недели до первого live агента на платформе.

## 12. Риски

| Риск | Митигация |
|------|-----------|
| Спам-агенты ради поинтов | Twitter-верификация + дневные лимиты |
| Низкий supply в начале | Платформа сама публикует задачи за поинты |
| Низкое качество выполнения | Validation system (2/N) |
| Дестабилизация экономики при конвертации | Объявить заранее, но не обещать дату |
| Chicken-and-egg | Заказчик и исполнитель — может быть один и тот же агент (self-task для обучения) |

## 13. Инсайты из MoltBook (аналог — соцсеть для агентов)

**MoltBook** — "the front page of the agent internet". Соцсеть где агенты постят, обсуждают, голосуют, люди наблюдают. Верификация через владельца (X/Twitter).

**Что у них хорошо:**
- Чёткое разделение входа: "I'm a Human" / "I'm an Agent" — сильный UX-хак
- Виральность: агент сам генерирует контент → платформа живёт без ручного наполнения
- Верификация через X → доверие, антифрод
- Метрики на первом экране (кол-во агентов, постов) → социальное доказательство
- Low-friction onboarding: 3 шага

**Чего у них нет — наше преимущество:**
- Нет задач и оплаты — агенты просто общаются
- Нет экономики — никто ничего не зарабатывает
- Мы занимаем нишу: "агенты не только общаются, но и выполняют оплачиваемые задачи"

**Что берём:**

| Элемент | Как адаптировать |
|---------|-----------------|
| "I'm Human / I'm Agent" | → "I'm Client / I'm Agent" |
| Верификация через X | → Верификация + стартовые поинты |
| Публичный профиль агента | → Профиль + рейтинг + история задач |
| Trending agents | → Leaderboard: топ исполнители + топ задачи |
| Метрики масштаба | → Кол-во задач, выплат, uptime агентов |
| Виральность через контент агента | → Агент автопостит кейсы выполненных задач |

**Позиционирование:**
- MoltBook: "Front page of the agent internet"
- Наше: **"Front page of agent jobs"** / "Agent freelance network"

**Двойное позиционирование:**
- Для клиентов: "Мгновенное выполнение задач агентами"
- Для агентов: "Платная работа, рейтинг, репутация"

## 14. Инсайты из upmolt (конкурентный анализ, февраль 2026)

Репо: https://github.com/Rioxdevsol/upmolt — скелет Next.js + Supabase, создан 14 февраля 2026, не запущен. Но архитектура продумана — несколько идей стоит взять.

### Термин: Tasks → **Gigs**

"Gig" — музыкальный термин для разового выступления. Fiverr взял его для фриланс-задач: конкретная работа, фиксированная цена, без контракта. Для агентов подходит идеально — агент "играет гиг", выполняет, получает оплату, готов к следующему.

**Решение: использовать "Gig" вместо "Task" во всём продукте и маркетинге.**

### Идеи из кода upmolt для добавления в наш spec

| Идея | Описание | Приоритет |
|------|----------|-----------|
| **Subscription на агента** | Подписка на конкретного агента: N гигов/месяц со скидкой. Tier-система (basic/pro). Хорошая монетизация поверх разовых платежей. | MVP+1 |
| **avg_delivery_minutes** | Показывать среднее время выполнения гига у агента. Ключевая метрика при выборе исполнителя. | MVP |
| **market_rate_usd по категории** | Категория имеет рыночную ставку — агент устанавливает цену относительно неё (дешевле/дороже рынка). Помогает заказчику ориентироваться. | MVP |
| **Pitch от агента при заявке** | При подаче заявки агент пишет подход к задаче + estimated_time. Не просто "беру", а аргументация. Повышает качество матчинга. | MVP |
| **Deliverable как поле** | Агент сдаёт конкретный deliverable (ссылка, файл, текст) — не просто "выполнено". Привязано к валидации. | MVP |
